# Project Task List Page

**Specification:** Реализовать динамическую страницу списка задач проекта, которая открывается при клике на карточку проекта на главной странице. На странице должны отображаться карточки задач в стиле предоставленного макета: поиск, фильтры (чипы), кнопка добавления, различные состояния карточек (в работе, просрочено, на проверке, сделано) и пустое состояние. Страница должна подхватывать данные выбранного проекта и быть адаптивной, поддерживать светлую/тёмную тему.

## User Stories

- Как руководитель проекта, я хочу нажать на карточку проекта и увидеть все задачи этого проекта, чтобы быстро оценить статус работ.
- Как участник команды, я хочу фильтровать задачи по статусу и искать по названию/исполнителю, чтобы находить актуальные для меня задачи.
- Как руководитель проекта, я хочу создавать новые задачи со страницы проекта, чтобы не возвращаться на главную.

## Main scenarios and rules

- Клик по карточке проекта на главной странице открывает маршрут `/projects/[id]/tasks` (или аналогичный), где `id` — идентификатор проекта.
- Страница подгружает данные выбранного проекта (название, возможно аватар/описание) и список задач из API/стора. При отсутствии данных отображается состояние «Здесь пока нет задач».
- Над списком задач отображается поисковое поле и горизонтальный список фильтров-«чипов» по статусам: «Все», «В работе», «Просрочено», «На проверку», «Сделано». Активный фильтр подсвечивается.
- Карточка задачи показывает название, исполнителя (аватар и имя), срок/признак просрочки, статус (цветной бейдж), приоритет. Завершенные задачи отображаются полупрозрачными и зачёркнутыми.
- Предусмотреть кнопку добавления задачи в хедере и плавающую кнопку действия внизу (для мобильной версии). Нажатие инициирует открытие модального окна/маршрута создания задачи (точная реализация — в рамках отдельной задачи, если не существует).
- Поддержка тёмной темы: цвета и фон соответствуют макету, использовать переменные/классы Tailwind с поддержкой `dark:`.
- Должна быть адаптивная верстка: чипы скроллятся горизонтально, карточки растягиваются на ширину экрана, элементы управления доступны на мобильных устройствах.
- Обновление фильтров и поиска не перезагружает страницу, работает на клиенте (состояние в сторе/компоненте).

## Non-functional requirements

- Роутинг и загрузка данных должны работать без полной перезагрузки страницы (SPA). Время отклика при фильтрации/поиске не более 200 мс при объёме до 200 задач.
- Обеспечить доступность: элементы управления должны иметь aria-метки, фокусное состояние, достаточный контраст.
- Страница должна корректно отображаться на экранах от 320px, поддерживать локализацию на русском (строки в переводах, если используется i18n).
- Код должен быть покрыт минимум smoke-тестом/юнит-тестом для логики фильтрации (если такая практика уже есть) и пройти линтер/типизацию проекта.

## Assumptions

- Данные по проекту и задачам доступны через существующий API/стор. Если нет, потребуется временный мок или дополнительная задача по API.
- Механизм создания задачи (модалка/страница) уже реализован; если нет — нужно уточнить отдельную задачу.
- Карточка проекта на главной странице содержит идентификатор/ссылку, позволяющую маршрутизировать на страницу задач.
- Требования к дизайну соответствуют предоставленному HTML-примеру; допускается адаптация под существующую дизайн-систему.
