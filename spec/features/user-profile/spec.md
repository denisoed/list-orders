# User Profile Public Page

**Specification:** Создать динамическую страницу публичного профиля коллеги, доступную по клику на пользователя со страницы списка коллег, отображающую основные сведения, контактную информацию и назначенные задачи с историей в стилистике текущего приложения.

## User Stories

- Как сотрудник отдела, просматривающий список коллег, я хочу переходить на публичный профиль выбранного пользователя, чтобы видеть актуальные контакты и статус его задач.
- Как руководитель, я хочу на странице профиля видеть распределение задач по статусам, чтобы быстро оценивать загрузку подчинённого.
- Как любой пользователь системы, я хочу, чтобы профиль корректно отображался на мобильных и настольных устройствах, чтобы пользоваться им независимо от устройства.

## Main scenarios and rules

- Страница доступна по динамическому маршруту (например, `/colleagues/[id]`) и открывается по нажатию на карточку пользователя в списке коллег.
- Заголовок страницы содержит имя и роль пользователя, аватар и кнопку возврата/закрытия, соответствующие компонентам навигации приложения.
- Блок контактов показывает доступные каналы связи (мессенджер, телефон, email и др.) с иконками из общей библиотеки и соблюдением светлой/тёмной темы.
- Секция задач содержит фильтры по статусам и списки задач, распределённые на активные и исторические, используя существующие компоненты карточек и токены дизайна.
- Пустые состояния (нет задач или контактов) отображаются дружественными сообщениями и иконографикой, принятой в приложении.
- Данные подгружаются асинхронно на стороне клиента или сервера через существующий API профилей; в ожидании отображаются скелетоны или индикатор загрузки.
- Доступ к странице ограничен авторизованным пользователям; при ошибке загрузки отображается стандартный блок ошибок.

## Non-functional requirements

- Поддержка адаптивной вёрстки с сохранением сетки и отступов приложения (Tailwind конфигурация проекта).
- Соответствие светлой и тёмной темам, использование глобальных токенов цвета и типографики.
- Рендеринг профиля должен укладываться в SLA загрузки страницы приложения (<=2 секунды на типовом соединении) при наличии кеширования запросов.
- Код должен быть покрыт модульными/компонентными тестами для критичных частей (рендер секций, переключение фильтров).
- Использовать доступные элементы управления с ARIA-атрибутами и возможностью навигации с клавиатуры.

## Assumptions

- API для получения данных пользователя и его задач уже существует и предоставляет необходимые поля (контакты, роли, список задач со статусами).
- В приложении есть общий Layout и компоненты карточек, которые можно переиспользовать для секций профиля.
- Не требуется редактирование данных на публичной странице — она предназначена только для просмотра.
- Требуется уточнение, какие статусы задач поддерживаются и какие фильтры нужно отобразить по умолчанию.
